# ManoSathi Mental Health Platform - Requirements

## 1. Overview

ManoSathi is a comprehensive mental health support platform designed to provide accessible, culturally-sensitive therapeutic support to users across India. The platform combines AI-powered conversational therapy, journaling with intelligent analysis, mood tracking, crisis intervention features, and educational resources — all powered entirely by AWS cloud infrastructure.

**Target Users:** Young adults (18-35) in India seeking mental health support, stress management, and emotional wellness tools.

**Core Value Proposition:** Provide 24/7 accessible, culturally-aware mental health support through AI-powered conversations (Amazon Bedrock), personalized insights, evidence-based therapeutic techniques, and clinically validated resources — with zero dependence on third-party AI platforms.

## 2. User Stories & Acceptance Criteria

### 2.1 Authentication & User Management

#### 2.1.1 User Registration
**As a** new user  
**I want to** create an account with email and password  
**So that** I can access personalized mental health support

**Acceptance Criteria:**
- User can sign up with email and password
- Password must be at least 8 characters
- User profile is automatically created in DynamoDB (manosathi-users) upon registration
- JWT token is returned and stored locally for session persistence
- Email validation is performed before account creation
- Error messages are clear and actionable
- Duplicate email detection with helpful messaging

#### 2.1.2 User Authentication
**As a** returning user  
**I want to** sign in to my account securely  
**So that** I can access my previous conversations and journal entries

**Acceptance Criteria:**
- User can sign in with email/password via AWS API Gateway + Lambda
- JWT token generated by authLambda is stored in localStorage
- Session automatically restored on page reload via `/auth/me` endpoint
- Protected routes redirect unauthenticated users to login
- Token expiry triggers automatic signout via session timeout hook
- Session timeout configured for HIPAA compliance

#### 2.1.3 User Onboarding
**As a** new user  
**I want to** complete an onboarding questionnaire  
**So that** the AI can provide personalized support

**Acceptance Criteria:**
- Onboarding flow collects: age range, student/working status, primary stressors, preferred language
- User cannot access main features until onboarding is complete
- Onboarding data is stored in user profile in DynamoDB
- User can update preferences later in settings
- Onboarding completion status tracked via `onboardingComplete` flag

### 2.2 AI Companion Chat

#### 2.2.1 Conversational Therapy
**As a** user  
**I want to** chat with an AI companion about my feelings and concerns  
**So that** I can receive empathetic support and guidance

**Acceptance Criteria:**
- User can send text messages to the AI companion
- AI responds with empathetic, culturally-aware messages in Hindi/English/mixed
- Powered by Amazon Bedrock (amazon.nova-pro-v1:0 via ConverseCommand)
- Conversation history is maintained within session (last 30 messages)
- AI uses hybrid memory strategy: all user messages + last 2 AI responses
- AI adapts communication style based on user profile stored in DynamoDB
- Messages display with timestamps
- Loading indicator shown while AI generates response
- Support for markdown formatting (bold, bullet points)
- Automatic retry (up to 3 attempts) on Lambda cold start or network errors

#### 2.2.2 Crisis Detection & Intervention
**As a** user in distress  
**I want** the system to recognize crisis situations  
**So that** I can receive immediate help and resources

**Acceptance Criteria:**
- System detects crisis keywords and sentiment in real-time
- Crisis levels: none, low, moderate, high, severe
- High/severe crisis triggers immediate helpline information
- Crisis resources include AASRA (91-22-27546669) and other Indian helplines
- AI provides supportive crisis-specific responses via Bedrock
- User can continue conversation after crisis intervention
- Risk assessment level is returned in every AI response payload

#### 2.2.3 Chat Session Management
**As a** user  
**I want to** manage multiple chat conversations  
**So that** I can organize different topics or time periods

**Acceptance Criteria:**
- User can create new chat sessions
- Sessions stored in DynamoDB (manosathi-chats) with userId GSI
- User can view list of previous chat sessions with preview and timestamp
- Sessions display last message timestamp and message count
- User can rename chat sessions via PUT /sessions/{id}
- User can delete chat sessions via DELETE /sessions/{id}
- Current session highlighted in sidebar
- Mobile-responsive sidebar with overlay

#### 2.2.4 Long-Term Memory System
**As a** user  
**I want** the AI to remember important facts about me across sessions  
**So that** I don't have to repeat myself

**Acceptance Criteria:**
- System extracts and stores key facts from conversations using rule-based pattern matching
- Fact categories: education, family, career, health, interests, challenges, goals
- Memory updated after each conversation via memoryManager
- Memory stored in DynamoDB manosathi-users table with `memory_{userId}` key
- AI retrieves top 5 most important facts for context in new conversations
- Facts persist across chat sessions with merge and deduplication logic

### 2.3 Journal Feature

#### 2.3.1 Journal Entry Creation
**As a** user  
**I want to** write daily journal entries with mood tracking  
**So that** I can reflect on my thoughts and emotions

**Acceptance Criteria:**
- User can create new journal entries with title and content
- User selects mood: Amazing, Good, Okay, Low, Rough (with emojis)
- Entry content supports multi-line text
- Character count displayed while typing
- Entries are saved to DynamoDB (manosathi-journals) via POST /journals
- User receives confirmation when entry is saved

#### 2.3.2 AI Journal Analysis
**As a** user  
**I want** detailed AI-powered insights on my journal entries  
**So that** I can understand my thought patterns and receive therapeutic guidance

**Acceptance Criteria:**
- AI analysis triggered asynchronously (fire and forget) when entry is created
- Analysis powered by Amazon Bedrock via /journal/analyze endpoint
- Analysis generates the following specific sections:
  - **Conversational Response:** Warm, empathetic feedback from ManoSathi
  - **Therapeutic Perspective:** Professional psychological insight
  - **Thought Patterns:** Identification of cognitive distortions or patterns
  - **Emotional Nuance:** Deep emotional analysis beyond surface mood
  - **Try This:** Gentle, actionable suggestion or exercise
  - **Key Themes:** List of recurring topics or themes
- Analysis is displayed with specific icons and color-coded cards
- Analysis supports multiple languages (Hindi, English, etc.)
- Analysis result is stored back into the journal entry in DynamoDB

#### 2.3.3 Journal History & Management
**As a** user  
**I want to** view, edit, and manage my past journal entries  
**So that** I can track my emotional journey over time

**Acceptance Criteria:**
- User can view list of past journal entries (newest first) via GET /journals
- Each entry shows: mood emoji, title, date, content preview, AI insights
- User can edit existing entries via PUT /journals/{id}
- User can delete journal entries via DELETE /journals/{id} with confirmation
- Entries polled every 3 seconds for real-time-like updates

## 3. Non-Functional Requirements

### 3.1 Performance
- AI responses generated within 5 seconds (warm Lambda)
- Lambda cold start handled by 3-attempt retry with backoff in frontend
- Journal analysis completed asynchronously (fire and forget)
- Polling every 3 seconds for journal and session updates

### 3.2 Security
- All user data encrypted in transit (HTTPS via API Gateway)
- JWT-based authentication (generated and verified by authLambda)
- DynamoDB access restricted via IAM roles (Lambda execution role)
- Admin routes protected with role-based access control
- Sensitive data (API keys) stored in AWS Lambda environment variables
- Session timeout hook enforces HIPAA-compliant auto-logout

### 3.3 Scalability
- AWS Lambda auto-scales on demand (no instance cap)
- DynamoDB on-demand billing (PAY_PER_REQUEST)
- API Gateway handles burst traffic automatically
- Amazon Bedrock scales with AWS capacity

### 3.4 Reliability
- Error handling with user-friendly messages
- Fallback responses if Bedrock fails after 3 retries
- Retry logic with exponential backoff in both frontend and Lambda
- Graceful degradation when services are unavailable
- Offline support for critical features (Resources, Quotes)

### 3.5 Usability
- Intuitive navigation with clear visual hierarchy
- Consistent design language (Tailwind CSS)
- Accessibility-compliant UI components
- Loading states for all async operations
- Toast notifications for user feedback

### 3.6 Cultural Sensitivity
- AI (Amazon Bedrock Nova Pro) prompted with Indian cultural context
- Respectful of diverse religious and social backgrounds
- Language-appropriate communication styles
- Crisis resources specific to India (AASRA, iCall, Vandrevala Foundation)

## 4. Technical Constraints

### 4.1 Technology Stack
- **Frontend:** React 18.3, TypeScript 5.9, Vite 6.3
- **Styling:** Tailwind CSS 3.4
- **Backend:** AWS Lambda (Node.js 20.x) via Serverless Framework 3
- **Database:** Amazon DynamoDB (manosathi-users, manosathi-chats, manosathi-journals, manosathi-treatments)
- **API:** Amazon API Gateway (HTTP API)
- **Authentication:** Custom JWT via authLambda (no Cognito)
- **AI:** Amazon Bedrock (amazon.nova-pro-v1:0, ConverseCommand SDK)
- **Memory:** DynamoDB-backed pattern extraction (memoryManager)
- **Mobile:** Capacitor 8.0

### 4.2 Browser Support
- Modern browsers (Chrome, Firefox, Safari, Edge)
- Mobile browsers (iOS Safari, Chrome Mobile)

### 4.3 API Dependencies
- AWS SDK v3 (@aws-sdk/client-bedrock-runtime, @aws-sdk/client-dynamodb, @aws-sdk/lib-dynamodb)
- Serverless Plugin TypeScript 2.1.5
- Quotable API (for inspirational quotes, with local fallback)

### 4.4 AWS Services Used
- **Amazon Bedrock** — Foundation model access (Nova Pro) for empathic and clinical AI responses
- **AWS Lambda** — Serverless compute for all backend functions
- **Amazon API Gateway** — HTTP API with CORS + JWT proxy
- **Amazon DynamoDB** — Primary NoSQL database (users, chats, journals, treatments)
- **Amazon CloudWatch** — Lambda logging and monitoring
- **AWS IAM** — Role-based access control for Lambda → DynamoDB → Bedrock
